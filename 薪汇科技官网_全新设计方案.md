# 薪汇科技官网全新设计方案

> **项目目标**：将 xhmxb.com 从"民薪保 HR SaaS"改版为"现代科技数据服务商官网"
> **核心场景**：手机端通过微信服务号菜单点击跳转 M-WEB
> **参考站点**：shuxin-tech.com（深圳数鑫科技）
> **设计原则**：移动优先、现代科技感、内容可管理、快速上线

## 二、数据库选型评审

这是一个关键决策点。以下从第一性原理分析三个候选方案：

### 2.1 候选对比

| 维度 | SQLite | PostgreSQL | MySQL |
|------|--------|------------|-------|
| **适用场景** | 嵌入式/本地/单机应用 | 通用生产级数据库 | 通用生产级数据库 |
| **并发能力** | 单写者，不适合 Web 并发 | 优秀的 MVCC 并发控制 | 良好 |
| **JSON 支持** | 基础 | 原生 JSONB，可索引可查询 | JSON 类型但功能弱于 PG |
| **全文搜索** | 需 FTS5 扩展 | 内置 tsvector/tsquery | 需 FULLTEXT 索引 |
| **部署复杂度** | 零配置，文件即数据库 | 需要独立服务进程 | 需要独立服务进程 |
| **备份恢复** | 拷贝文件即可 | pg_dump / 流复制 | mysqldump |
| **生态成熟度** | 嵌入式领先 | 最活跃的开源 RDBMS | 广泛使用 |
| **类型系统** | 弱类型（动态） | 强类型，丰富的数据类型 | 中等 |
| **扩展性** | 无 | 丰富的扩展生态 | 有限 |

### 2.2 结论：推荐 PostgreSQL

**理由（从第一性原理）：**

1. **并发是硬需求**：官网是多用户同时访问的 Web 应用，后台有管理员同时编辑内容。SQLite 的单写者锁机制在并发场景下会成为瓶颈，直接排除。

2. **JSONB 对 CMS 有天然优势**：CMS 的公司信息（cms_company_info）本质上是 Key-Value 结构，PostgreSQL 的 JSONB 字段可以直接存储和查询，比 MySQL 更自然更高效。

3. **全文搜索内置**：文章搜索不需要引入额外的 Elasticsearch，PostgreSQL 自带的全文搜索足够应对 CMS 文章量级。

4. **现代技术栈的首选**：Strapi、Payload CMS、Directus 等主流 Headless CMS 都将 PostgreSQL 作为推荐数据库。

5. **不选 MySQL 的原因**：既然是全新设计不考虑历史技术栈，PostgreSQL 在功能、标准遵从性、扩展性上全面优于 MySQL 5.7。没有迁移负担的情况下，没有理由选择 MySQL。

6. **不选 SQLite 的原因**：SQLite 适合嵌入式场景（移动 App、桌面软件、IoT），不适合作为生产 Web 应用的主数据库。唯一的优势（零配置）不足以弥补并发和功能的不足。

> **最终决策：PostgreSQL 16+**

## 三、技术架构（全新设计）

### 3.1 整体方案：Headless CMS + SSR 前端

既然可以使用成熟的 CMS 系统，推荐采用 **Headless CMS** 架构。理由：

- 后台管理面板开箱即用（文章编辑、Banner 管理、图片上传、权限控制全都有）
- 前后端解耦，前台可以独立做 SSR/SSG
- 不用重复造轮子，大幅缩短开发周期

### 3.2 方案对比

| 方案 | 前台 | 后台CMS | 数据库 | 开发量 | 说明 |
|------|------|---------|--------|--------|------|
| **A：Strapi + Nuxt 3** | Nuxt 3 (Vue) | Strapi v5 | PostgreSQL | 小 | Vue 生态，团队有 Vue 经验 |
| **B：Payload + Next.js** | Next.js (React) | Payload CMS 3.0 | PostgreSQL | 小 | TypeScript 全栈，最现代 |
| **C：全自研** | Nuxt 3 / Next.js | 自建 Admin | PostgreSQL | 大 | 完全可控，但开发量翻倍 |

### 3.3 推荐方案 A：Strapi v5 + Nuxt 3

**选择理由：**

1. **Strapi v5** 是目前最成熟的开源 Headless CMS，社区活跃，文档完善
2. **Nuxt 3** 基于 Vue 3，团队有 Vue 经验（历史系统均为 Vue），学习成本最低
3. Strapi 后台面板自带富文本编辑器、图片管理、用户权限，不需要额外开发
4. Strapi 支持自定义认证策略，可以实现邮箱验证码登录

### 5.3 邮箱验证码登录方案

Strapi 默认使用邮箱+密码登录。需要自定义实现邮箱验证码登录：

```
登录流程：
1. 管理员输入邮箱（限 @minxinhui.com 域名）
2. 后端校验域名 → 生成 6 位验证码 → 存 Redis（5分钟 TTL）→ 发邮件
3. 管理员输入验证码
4. 后端校验验证码 → 签发 JWT Token → 返回
5. 前端存储 Token，后续请求携带 Authorization Header

实现方式：
- Strapi 自定义 Plugin 或 Custom Controller
- 路由：POST /api/auth/send-code、POST /api/auth/verify-code
- 依赖：nodemailer + ioredis
```

```
│  is_read      │     │  phone           │
│  created_at   │     │  email           │
└───────────────┘     │  icp_number      │
                      │  wechat_qr_id    │
                      └──────────────────┘
```

> **注**：使用 Strapi 时，表结构由 Strapi 自动生成和管理，上面是逻辑模型。实际的物理表名和字段名会遵循 Strapi 的命名规范。

### 6.2 PostgreSQL 特有优势的利用

```sql
-- 文章富文本内容使用 JSONB 存储（Strapi Blocks 格式）
-- 可以直接在数据库层面做全文搜索
CREATE INDEX idx_article_content_search
ON cms_articles USING gin(to_tsvector('simple', title || ' ' || summary));

-- 公司信息的灵活扩展字段
-- 未来可以直接在 JSONB 中加新字段，无需变更表结构
ALTER TABLE cms_company_info ADD COLUMN extra_info JSONB DEFAULT '{}';
```

## 八、部署架构

### 8.1 Docker Compose 一键部署

```yaml
# docker-compose.yml
services:
  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf
      - ./ssl:/etc/nginx/ssl
    depends_on:
      - nuxt
      - strapi

  nuxt:
    build: ./xinghui-web
    environment:
      - STRAPI_URL=http://strapi:1337
    ports:
      - "3000:3000"

  strapi:
    build: ./xinghui-cms
    environment:
      - DATABASE_CLIENT=postgres
      - DATABASE_HOST=postgres
      - DATABASE_PORT=5432
      - DATABASE_NAME=xinghui_cms
      - DATABASE_USERNAME=xinghui
      - DATABASE_PASSWORD=${DB_PASSWORD}
      - REDIS_HOST=redis
    ports:
      - "1337:1337"
    depends_on:
      - postgres
      - redis

  postgres:
    image: postgres:16-alpine
    environment:
      - POSTGRES_DB=xinghui_cms
      - POSTGRES_USER=xinghui
      - POSTGRES_PASSWORD=${DB_PASSWORD}
    volumes:
      - pgdata:/var/lib/postgresql/data
    ports:
      - "5432:5432"

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"

volumes:
  pgdata:
```
